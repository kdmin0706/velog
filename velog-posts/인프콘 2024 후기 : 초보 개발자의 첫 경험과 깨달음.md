<h2 id="3-세션">3. 세션</h2>
<h3 id="3-1-인프런-아키텍처-2024">3-1. 인프런 아키텍처 2024</h3>
<p><strong>발표자 : 인프랩 CTO 이동욱님</strong></p>
<p>매년 인프콘에서 인프런 아키텍처의 변화에 대해서 발표를 해주신 거 같다.
올해는 인프런의 <code>국제화</code>를 위해 이전 아키텍처에서 각 목적 별로 기능을 개발하고 성능을 보완하는 내용을 발표해주셨다.</p>
<p><img alt="" src="https://velog.velcdn.com/images/kdmin0706/post/d2e5684c-65c8-4d34-a65e-2fc89e909714/image.png" /></p>
<p><strong>글로벌화를 위한 트래픽(이미지) 비용 개선</strong></p>
<blockquote>
<p>기존 이미지 저장(as is)
cloud Front(캐싱) -&gt; Lamda Storage(리사이징) -&gt; S3(원본 보관)</p>
</blockquote>
<p>위의 문제점은 사용자가 들어올 때마다 비용은 발생하는데 사용자가 결제하지 않으면 비용을 회수할수 없다고 한다.</p>
<blockquote>
<p>변경 방법(to be)
avif(품질 저하 없는 고효율 압축)를 사용해 Lamda Storage에서 avif로 파알 변환을 하여 S3에 저장한다.</p>
</blockquote>
<p><img alt="" src="https://velog.velcdn.com/images/kdmin0706/post/8ac89bdd-8408-4ab1-bba2-ba616f76db17/image.png" /></p>
<p>그림과 함께 설명을 들으면서 어떤 과정으로 진행했는지 이해하는 건 어렵지 않았는데, 막상 세션을 듣고 나니 AWS에 대한 지식이 많이 부족하고 있다는 것을 새삼느끼게 되었다.</p>
<p>마지막에 향로님이 말하신 <code>펠리컨적 사고</code> 도전해보고 이를 바탕으로 더욱 나은 프로덕트를 구성하고 제공할 수 있도록 하는 것이 중요하다는 것을 다시금 깨우친 거 같다.</p>
<h3 id="3-2-경력이-늘수록-cs-이론이-중요해지는-이유">3-2. 경력이 늘수록 CS 이론이 중요해지는 이유</h3>
<p><strong>발표자 : <a href="https://www.youtube.com/@nullnull_not_eq_null">널널한 개발자</a> 최호성님</strong></p>
<p>이번 세션에서는 CS 이론에 대한 내용이 주를 이를 줄 알았는데, <code>고졸 비전공자</code>인 내 입장에서 올해 인프콘에서 가장 인상 깊었던 강연이었던 거 같다. </p>
<p><img alt="" src="https://velog.velcdn.com/images/kdmin0706/post/d7e215be-cb08-44b9-af7a-397ab6aa4cae/image.png" /></p>
<p>CS 이론의 중요성을 다양한 사례를 통해 설명해주셨습니다. </p>
<p>비전공자/국비캠프/SI 출신 5년차 개발자의 물경력에 대한 상황과 신입 개발자를 준비 중인데 코테/CS/각종 프레임워크를 공부했는데 아직도 갈피를 잡지 못하는 상황에서의 상황에 대한 예시를 활용하여 위의 개발자들의 어려움을 cs 지식을 바탕으로 설명을 해주셨다.</p>
<p><img alt="" src="https://velog.velcdn.com/images/kdmin0706/post/f52387e4-e256-4b73-9ccb-61c8f0048309/image.png" /></p>
<p>예를 들어 컴퓨터 구조의 <code>Stack</code>에 대해서 공부한다면 자료구조/알고리즘에 있는 <code>Stack</code>에 대해 공부를 할 것이고 예로 Java/Spring 진영이라면 <code>stack</code>을 활용한 프로그래밍 언어에 대해 공부를 해야할 것이다. 이를 보면 cs와 프로그래밍 언어는 이해관계에 있다고 생각이 된다.</p>
<p><strong>세션 마지막 아래 내용을 보면서 고민을 많이 하게된 거 같다</strong></p>
<ol>
<li><p>cs 공부를 하지 않으면 결국 개발자로 살아가면서 하는 후회</p>
<ul>
<li>모든 개발자에게는 <code>cs 이론</code>이 중요(성능, 장애대응, 설계)</li>
<li>최초 <code>나는 왜 개발자가 된 것</code>인지 다시 생각해볼 것(취업, 밥벌이, 개발자)</li>
<li>분야에 따른 차등을 인정하더라도 <code>최소한의 지식은 필수</code>(더 잘해내고 싶다면)</li>
</ul>
</li>
<li><p>목표에 대해 진지하게 다시 생각해볼 것</p>
<ul>
<li>애초에 <code>나는 어떤 이유로 개발자가 되려는 것</code>인지 스스로에게 질문</li>
<li>앞으로 <code>얼마나 올랫동안</code> 개발자로 살아갈 것인가?</li>
<li>다양한 분야에 대해 알아가고 자기 분야에 대한 <code>독보적인 전문성 확보</code></li>
</ul>
</li>
</ol>
<h3 id="3-3-혹시-당신은-데이터를-모르는-개발자인가요">3-3. 혹시 당신은 데이터를 모르는 개발자인가요?</h3>
<p><strong>발표자 : 아임웹 김지호님</strong></p>
<p>발표 제목을 보고 백엔드 개발자로써 Database에 대한 중요성을 알려주겠거니하고 세션을 들었는데, 막상 내용은 달랐던 거 같다. 현재 백엔드와 데이터 엔지니어를 겸직하는 개발자이셔서 서로의 관점에서 어떻게 생각할지, 협업을 위해서는 어떻게 진행하는 것이 올바르게 하는 것인지 인지할 수 있도록 세션을 구성하신 거 같다.</p>
<p>데이터를 보는 시각으로 구분을 해보자면 백엔드 엔지니어는 <code>행</code>을 읽고, 데이터 엔지니어는 <code>열</code>을 읽다는 것이다.</p>
<p>또한 DB는 단순히 데이터만 저장하는 곳이 아니라는 것이다. 데이터의 히스토리를 파악하고, 분석할 때도 사용되고, 분석환경은 라이브 환경에서 사용되는 DB와 별개로 분리해서 관리된다는 것이다.</p>
<p>마지막으로 <code>데이터 무결성</code>에 대한 내용이다.
데이터 무결성에는 3가지가 존재하는데 <code>유효성</code>, <code>정확성</code>, <code>일관성</code>이다.
유효성의 경우에는 데이터 범위 내에서 존재하는지 확인하는 것이고, 데이터의 정확성은 실제 값을 정확하게 나타내는가에 대한 내용인데 여기에는 <code>중복과 누락</code>이 포함된다. 즉, <code>Delete</code> 쿼리를 사용할 때 주의해라라는 내용을 포함하라는 것이었다. 
데이터 일관성의 경우는 하나의 컬럼에 하나의 맥락이 들어가있냐이다. 1개의 컬럼에는 1개의 값만 변경해야된다. 즉, 여러 컬럼에 영향이 미치면 안된다는 내용이다.</p>
<h3 id="3-4-클린-스프링-스프링-개발자를-위한-클린코드">3-4. 클린 스프링: 스프링 개발자를 위한 클린코드</h3>
<p><strong>발표자 : 토비 이일민님</strong></p>
<p><strong>클린 코드를 지향할수록 점점 구현능력이 떨어진다?</strong>
이는 좋은 코드에 대한 고민이 많아질수록 속도가 떨어지는 현상으로, 이로 인해 필요한 기능을 제때 구현하지 못하는 경우가 발생한다. 이는 좋은 코드, 구조, 설계에 대한 집착의 원인이 될 수 있다.</p>
<p><img alt="" src="https://velog.velcdn.com/images/kdmin0706/post/07e053c8-da33-4ff3-9a94-7218aa8f4ee8/image.png" /></p>
<blockquote>
<p><strong>Clean code that Works</strong>✨
이번 세션의 주제를 한 마디로 정의한 단어를 통해 세션에서 전달하고자 하는 내용에 대해 알 수 있었다.</p>
</blockquote>
<p><code>작동하지 않는 클린코드</code>는 다음과 같이 구성된다.</p>
<ol>
<li>주석 미작성</li>
<li>리팩토링을 하지 않는다.</li>
<li>테스트 코드를 작성하지 않는다.</li>
<li>일정을 준수하지 않는다.</li>
<li>클린코드 원칙에 위배된다는 이유로 코드 리뷰를 진행하지 않는다.</li>
</ol>
<p>위의 내용과 대비되는 <code>작동하는 클린코드</code>는 다음과 같다.</p>
<ol>
<li>읽기 좋은 코드</li>
<li>이해하기 좋은 코드</li>
<li>확장성이 좋은 코드</li>
<li><strong>유지보수성</strong>이 좋은 코드 -&gt; 위의 3가지를 모두 포함하는 말이다.</li>
</ol>
<blockquote>
<p>즉, 클린코드가 강조하는 것은 유지보수성(maintainability)이다.</p>
</blockquote>
<p><strong>만들면서 배우는 클린 아키텍처</strong></p>
<ol>
<li>품질 요구 사항(비기능적 요구사항)에서 유지보수성을 특별하다.</li>
<li>유지보수하기 좋은 코드는 확장하기 좋고, 안전하고, 신뢰할 수 있고, 좋은 성능으로 발전시킬 수 있고, 상호 호환성이 뛰어나서 변경하기 쉽다.</li>
<li><strong>유지 보수성 == 변경가능성</strong></li>
</ol>
<p>유지보수하기 쉬운 클린 코드는 <code>유지보수성 &lt;-- 코드 --&gt; 생산성</code> 이다.
유지보수성과 개발 생산성은 서로 영향을 주는 관계라는 것이다.</p>
<p>만약, 개발생산성과 유지보수성 중 한쪽에라도 치우치게 된다면 다른 쪽에 영향을 주기 때문에 어느 한쪽으로 치우치지 않은 코드를 작성해야한다.</p>
<blockquote>
<p>유지보수성 &lt;- 코드 ------&gt; 생산성
코드가 유지보수성에 치우친 경우 생산성을 떨어진다.
유지보수성 &lt;----- 코드 -&gt; 생산성
코드가 생산성에 치우친 경우 유지보수성을 떨어진다.</p>
</blockquote>
<p><strong>기술 부채(Dept)</strong></p>
<ul>
<li>개발하는 소프트웨어에 대한 현재 이해를 반영하는 코드를 작성하고 <code>빠르게</code> 출시</li>
<li>소프트웨어에 대해서 배우게 된 것을 <code>리팩터링</code>을 통해 코드에 반영</li>
<li>그러지 않으면 이자가 쌓여서 점점 큰 부담</li>
</ul>
<p>이에 따른 유의사항은 아래와 같다.</p>
<ol>
<li>코드를 대충 작성하라는 것이 아니다.</li>
<li>부채는 나쁜 코드에 대한 변명이 될 수 없다.</li>
<li>처음부터 리팩터링하기 좋은 코드를 만들어야한다.</li>
</ol>
<p>정리하면 부채가 <strong>지속적으로 효과를 내려면 리팩터링(부채상환)이 동력이 되야</strong>한다는 것이다.</p>
<p>그렇다면 <code>리팩토링 하기 좋은 코드란 뭘까?</code></p>
<blockquote>
<p>바로 <code>테스트 코드</code>이다. 테스트 코드는 초기 개발 단계에서 시간이 더 걸릴 수 있으나 개발기간이 길어질수록 더욱 유리하기 때문이다.</p>
</blockquote>
<p>여기서 간단한 토비님의 Q&amp;A 예시 내용이다.</p>
<blockquote>
<p>Q : 회사에서 테스트 코드를 짜다가 구현하기 바쁜데 뭐하냐면서 혼났어요.
A(토비) : 테스트 코드를 빠르게 잘 만들면 되요!</p>
</blockquote>
<p><strong>클린 코드의 선순환</strong>
<img alt="" src="https://velog.velcdn.com/images/kdmin0706/post/5aa7d6a8-6294-464f-9619-046a9d085177/image.png" /></p>
<p>클린코드는 <code>리팩토링</code>을 통해 선순환을 반복한다.
유지보수성이 좋은 코드는 변경 가능성이 높다 🔁 빠르게 변경되는 코드는 개발 생산성이 좋다.</p>
<p>그렇다면 <strong>개발시작은 빠르게 간단하게 하려면?</strong></p>
<ol>
<li>익숙한 기술로</li>
<li>핵심 기능이 있는</li>
<li>동작하는</li>
<li>가장 단순한 코드를</li>
<li>리팩토링 하기 좋게 작성</li>
</ol>
<p><strong>일부 기능을 완벽하는 만드는 것으로 시작하는 것이 아니</strong>라는 것은 명심해야한다.</p>
<blockquote>
<p>균형을 잡기 위한 리팩토링을 잘하려면? -&gt; 테스트는 반드시 필요하다💡
테스트를 빠르고 효과적으로 작성하고 개발하는 능력을 키우려면? -&gt; 많은 연습이 필요하다💡</p>
</blockquote>
<p>클린 코드의 많은 원칙은 <code>팀원을 위해서</code> 나온 것이다.
즉, 상황에 따라서 다른 기준들을 가지고 있다는 것이다.</p>
<p><img alt="" src="https://velog.velcdn.com/images/kdmin0706/post/e6b58abe-31fd-49b7-ab1f-56db0cfb2883/image.png" /></p>
<p>마지막으로 토비님의 당부를 정리해보자면</p>
<p><strong>나만의 클린 코드는 없다</strong> 이기에 팀과 함께 결정하고 학습하고 <code>탐험</code>해야한다.
교양있는 개발자는 자신의 말을 하더라도 다른 사람의 기분을 나쁘게 하지 않는 사람이다.
쉽지는 않겠지만 함께 코드를 작성하면서 읽고 변경할 동료들에게 <code>친절</code>한 코드를 만들었으면 좋겠다.</p>
<h2 id="4-네트워킹-파티">4. 네트워킹 파티</h2>
<p><img alt="" src="https://velog.velcdn.com/images/kdmin0706/post/4dee0c97-7c75-48a2-bcec-f4cd36b54cdd/image.png" /></p>
<p>네트워킹 파티장에 들어가면 인프콘에 참여한 사람들의 이름이 써져있는 거대한 포토부스가 있었다.</p>
<p>네트워킹 파티장에는 사람이 엄청 많았고, 각 테이블마다 깃발이 있고 깃발에는 관심있는 주제에 대한 내용이 적혀져 있어서, 주제에 맞는 이야기를 하고 싶은 사람들끼리 모여서 대화를 나눴던 거 같다.</p>
<p>네트워킹 파티를 썩 기대하지 않았고, 그냥 잠깐 구경만 하다 가야겠다고 생각을 하던 와중에 한 쪽 주변에서 삼삼오오 사람들끼리 모이고 있길래 <code>개발 이모저모</code> 무리에 들어가서 대화를 했었다.
<img alt="" src="https://velog.velcdn.com/images/kdmin0706/post/c95b5270-081d-4728-93a7-11df9f2bd575/image.png" /></p>
<p>아까 세션을 들었었던 최호성님과 현업에서 일하고 계셨던 백엔드 개발자분들이랑 대화를 할 수 있었다. 막상 현업에서 일하고 계셨던 분들과 대화를 하면서 현재 취업 준비상황과 취업 방향에 대해서 물어보고, 어떤 식으로 준비하면 좋을 지에 대한 피드백도 받았다. 또한, 현재 개발자 시장 상황에 대해서 들으면서 이전과 달리 취업 문턱이 더욱 높아졌다는 생각을 실감하게 되었던 거 같았다. </p>
<h2 id="5-마치며">5. 마치며</h2>
<p>다른 세션 트랙들이 많았지만, 나의 부족한 실력과 대비해서 소화하기에는 버거울 수 있다고 생각되어 알짜배기로만 들으려고 했다. 막상 끝났을 때쯤 생각해보면 공부하려고 생각하지말고 지식을 얻자는 생각으로 편하게 들었으면 어땠을까 하는 아쉬움이 많았다. 그래도, 개발자를 꿈꾸고 처음 가보고 들어본 컨퍼런스 이기 때문에 오히려 더 성장에 대한 욕구를 얻을 수 있었던 계기였던 같다. 내년 인프콘에 올 수 있을지는 모르겠지만 온다면 취직을 해서 와보고 싶다!</p>
